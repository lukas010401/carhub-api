1. Principes d'architecture
Pour le MVP, l'architecture recommandee est un monolithe modulaire cote backend, avec un frontend
separe. Ce choix permet de garder une base de code lisible, rapide a mettre en production et
suffisamment propre pour evoluer vers plus de fonctionnalites sans repartir de zero.
- Frontend separe pour le SEO, la performance et l'experience utilisateur.
- Interface web responsive pour desktop, tablette et telephone des la V1.
- Approche mobile-first sur les composants critiques : header, recherche, listing cards, detail vehicule,
dashboard vendeur.
- Backend API separe pour centraliser la logique metier, la securite et les futures integrations.
- Base relationnelle PostgreSQL pour les annonces, les referentiels et les utilisateurs vendeurs.
- Stockage objet externe pour les images afin d'eviter de surcharger la base de donnees.
- Migrations EF Core pour versionner et faire evoluer le schema proprement.

3. Architecture frontend responsive
Le frontend Next.js doit etre pense comme une application web responsive des le depart. La V1 ne doit
pas etre seulement belle sur ecran PC : elle doit rester fluide et lisible sur smartphone, car beaucoup
d'utilisateurs consulteront les annonces sur telephone.
Structure recommandee
src/
app/
components/
features/
services/
lib/
types/
Exigences UX responsive
- Approche mobile-first pour les composants et pages publiques.
- Breakpoints clairs : mobile, tablette, desktop.
- Navigation adaptee : menu compact sur mobile, navigation complete sur desktop.
- Barre de recherche empilee sur mobile et horizontale sur desktop.
- Cartes vehicules sur 1 colonne mobile, 2 colonnes tablette, 3 a 4 colonnes desktop selon largeur.
- Page detail vehicule avec galerie en haut sur mobile et layout 2 colonnes sur desktop.
- Formulaires vendeurs simplifies sur mobile avec champs pleine largeur.
- Dashboard vendeur et admin utilisables sur mobile, avec tables empilees ou cartes si necessaire.

5. Pages et comportement responsive
- Homepage : hero, recherche rapide, dernieres annonces, marques populaires, categories populaires.
- Listing page : filtre en drawer ou bloc repliable sur mobile, sidebar fixe sur desktop.
- Detail annonce : CTA WhatsApp et Appeler toujours visibles sur mobile.
- Login/Register vendeur : formulaire centre, simple, lisible sur petit ecran.
- Creer annonce : progression claire, upload photo adapte tactile.
- Dashboard vendeur : cartes de statut compactes et liste d'annonces consultable sur mobile.
- Admin : priorite desktop, mais consultation mobile possible pour validation rapide.

9. Authentification, securite et workflow
- Consultation publique libre pour les visiteurs.
- Connexion obligatoire uniquement pour les vendeurs et les administrateurs.
- JWT + refresh tokens pour les sessions cote frontend.
- Ownership check sur les annonces : un vendeur ne modifie que ses propres annonces.
- Validation serveur systematique de tous les champs.
- Rate limiting sur les endpoints sensibles : login, upload, publication.
- Logs des actions admin et evenements critiques.
Workflow de publication d'annonce
- Le vendeur se connecte.
- Il cree une annonce au statut Draft.
- Il renseigne les champs et ajoute les images.
- Il soumet l'annonce : statut PendingReview.
- L'admin approuve ou rejette.
- Si l'annonce est approuvee, PublishedAt est renseigne et l'annonce devient visible publiquement.
10. Stockage des images
Les photos des vehicules ne doivent pas etre stockees dans PostgreSQL. La base ne conserve que les
metadonnees des images.
- Formats autorises : jpg, jpeg, png, webp
- Limite du nombre d'images par annonce
- Taille maximale par image
- Ordre d'affichage gere en base
- Compression ou redimensionnement plus tard si besoin
11. Recherche, performance et responsive
- Pour le MVP, PostgreSQL suffit pour les filtres principaux.
- Filtres : marque, modele, prix min/max, annee min/max, carburant, transmission, ville, categorie, tri.
- Tri : plus recent, prix croissant, prix decroissant, kilometrage croissant.
- Pagination obligatoire.
- Les pages de resultats doivent rester rapides sur reseau mobile.
- Images optimisees et lazy loading cote frontend pour de bonnes performances sur telephone.
- Meilisearch pourra etre ajoute plus tard si le volume augmente fortement.

13. Migrations et gestion du schema PostgreSQL
Le projet utilisera Entity Framework Core en Code First avec migrations. C'est le bon pattern pour suivre
l'evolution du schema et garder une base reproductible entre dev, staging et production.
Principe retenu
- DbContext, configurations EF et migrations dans AutoMada.Infrastructure.
- Entites et mappings definis dans le code, puis generation de migrations versionnees.
- Application du schema via migrations, pas a la main.
- Petites migrations lisibles plutot qu'enormes migrations difficiles a relire.
Structure recommandee
AutoMada.Infrastructure/
Persistence/
AppDbContext.cs
Configurations/
ListingConfiguration.cs
BrandConfiguration.cs
ModelConfiguration.cs
Migrations/
Workflow migrations
- Creer ou modifier les entites et configurations Fluent API.
- Generer une migration via le CLI EF Core.
- Appliquer la migration en local sur PostgreSQL.
- Commit la migration dans Git avec le code.
- En production, privilegier un script SQL ou une application controlee de la migration.
Commandes utiles
dotnet ef migrations add InitialCreate
dotnet ef migrations add AddListingCategory
dotnet ef database update
Bonnes pratiques
- Ne pas reecrire une ancienne migration deja partagee.
- Nommer clairement les migrations : InitialCreate, AddListingImages, AddPublishedAtToListings.
- Creer aussi les index utiles via les migrations.
- Eviter EnsureCreated pour un projet reel qui evolue.
Index importants a prevoir
- Listings(Status)
- Listings(BrandId)
- Listings(ModelId)
